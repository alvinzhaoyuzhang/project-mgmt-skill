# 任务分级标准:5 级层级

## 层级定义

```
🎯 目标         ← 项目最终成果 (1-3 个)
  🏁 里程碑      ← 关键阶段 / Gate / Sprint (3-7 个)
    📦 一级工作   ← 里程碑下主要工作包 (每里程碑 3-7 个)
      📋 二级工作  ← 叶子任务,可分配可估算 (每一级下 3-10 个)
        📌 三级工作 ← 仅复杂场景使用,不推荐
```

## 每级标准

### 🎯 目标

- **本质**:项目交付后的可验收成果
- **数量**:1-3 个
- **示例**:"完成 MVP 验收交付 · 三案例 3/3 跑通"
- **负责人**:项目经理兜底,不强制
- **拆到此为止的条件**:项目结束时可以回答"达成了吗 / 没达成哪条"

### 🏁 里程碑

- **本质**:关键阶段 / Gate / Sprint,通常对应一次对外交付或重大节点
- **数量**:3-7 个
- **示例**:`🚀 Sprint 1 开发冲刺(D1-D5)` / `✅ Sprint 2 数据验证+验收(D6-D10)` / `📦 已完成阶段`
- **负责人**:该阶段的主导人
- **命名**:突出**阶段语义**(Sprint/Phase)或**Gate 交付**(D5/D10),二选一即可
- **注**:本方案倾向"阶段即里程碑"(Sprint-as-milestone),是经验证最符合大多数用户心智的命名

### 📦 一级工作

- **本质**:里程碑下的**工作包**,对复杂里程碑进行一次切分
- **数量**:每个里程碑下 3-5 个(硬上限 7 个)
- **负责人**:**工作包 Lead(可选,不强制)**
  - 一级工作本身不干具体活,但可以指定"主线负责人"(如架构师、开发 Lead、数据 Lead)
  - 跟项目经理区分:项目经理在项目主表,负责整个项目;工作包 Lead 在任务表,只负责这个工作包
  - 没有明确 Lead 时留空,不要硬塞一个人

**一级工作的维度选择原则**(skill 级规则,不绑特定业务):

| 维度 | 适用场景 | 典型 📦 命名 |
|---|---|---|
| **模块维度** | 工程项目,按技术/业务模块切分 | 底层架构 / 具体功能实现 / 对外展示前端 / Gate |
| **角色维度** | 小团队分工明确,每个角色一个主线 | Owner 工作 / 开发工作 / 数据工作 / 规则工作 |
| **交付阶段维度** | 瀑布项目,按交付节奏切分 | 需求与设计 / 开发与联调 / 测试与验证 / 发布与归档 |

**选取规则**:

1. **维度统一**:同一个里程碑下的所有 📦 必须属于**同一个维度**,不要混用(别把"底层架构"和"Owner 工作"并列)
2. **跨里程碑可复用**:推荐同一项目里所有里程碑用相同的 📦 骨架;但**不强制每个里程碑都含全部 📦** —— 没有对应工作就省略该 📦,不要建空节点
3. **命名用名词短语**:"底层架构" / "对外展示前端"(✅);"做架构" / "前端开发中"(❌)
4. **数量 3-5 为佳**:<3 说明拆细意义不大(可直接挂 📋);>7 说明该升级为单独的里程碑或换维度

**"任务分类"维度应放在字段层,不放在层级层**:
- 任务分类(按角色/模块/业务域)用 `模块` 字段或自定义 `任务分类` 字段表达
- 然后通过**视图分组**实现跨维度浏览(层级是 top-down 的结构,字段分类是 orthogonal 的切片)

**本项目(AI 产品开发)的实际选择**:模块维度,5 个 📦:
- 底层架构 · 具体功能实现 · 对外展示前端 · 数据与规则 · Gate
- 这 5 个是**本项目的具体应用**,skill 不固化它们;后续其他项目按规则自选


**是否每个里程碑都需要 📦 一级**:

- 一般需要(提高层级一致性)
- **小型里程碑/只有几条叶子时可省略**,让 📋 二级直接挂 🏁 里程碑
- 但如果省略,该里程碑下**不要混用"有 📦 和无 📦"两种模式**(保持层级一致)

### 📋 二级工作(叶子任务)

**叶子任务硬标准**(必须同时满足):

1. **可估算** — 1-5 天能完成
2. **可分配** — 能指定唯一负责人
3. **可度量** — 有明确"交付物"字段描述验收标准

**反模式**(看到就拆):
- ❌ 一个任务 3 个并列负责人 → 拆成 3 个
- ❌ 预估超过 5 天 → 再拆一层
- ❌ 交付物写"完成 XX" → 写"接口 + 单测 + 文档"这类具体产物

### 📌 三级工作

- **用途**:仅在一级工作本身就很复杂、需要中间分组时使用
- **数量**:应尽量不用。实际项目大多数情况**不应出现**三级工作
- **反例**(不要这样做):
  - ❌ 把历史版本归档 / 里程碑证据类记录放到 📌 三级工作 —— 这些是**已完成的叶子任务**,应归为 📋 二级工作
  - ❌ 为了分类好看,硬加一层中间分组 —— 应用**视图分组**(by 模块/by 负责人)而非任务层级实现
- **真正的使用场景**(很少):当某个 📦 一级工作确实需要再拆一层中间组,且中间组下还有若干叶子时,才用 📌 三级工作

## 每层深度控制

**每层宽度规则**:3-7 个子项

- 少于 3:可能可以合并上去
- 多于 7:应该再拆一层 OR 重新设计分类

**深度控制**:
- 常规项目:`目标 → 里程碑 → 一级 → 二级` 四层足够
- 复杂项目:加 `三级工作`
- **不要超过 5 层**(迷失路径)

## 按项目场景选择层级骨架(不是看任务数,是看特性)

**5 档骨架,按项目特性对号入座**:

| 骨架 | 层级结构 | 适用场景 | 典型项目 |
|---|---|---|---|
| **扁平** | 📋 单层 | 任务少、无明显阶段、无明显模块划分 | 一次性收集 / 单人短期任务列表 |
| **目标导向** | 🎯 + 📋 | 有明确目标但执行并列,无阶段/模块划分 | 季度个人 OKR 跟进 |
| **时间导向** | 🎯 + 🏁 + 📋 | 项目**周期较长**,有明显阶段/Sprint/里程碑节点 | 3 个月以上的连续项目、按周期冲刺 |
| **模块导向** | 🎯 + 📦 + 📋 | 项目**模块/工作包多**但无明显时间阶段 | 平台类建设、多产品线并行 |
| **完整分级** | 🎯 + 🏁 + 📦 + 📋 | **既有时间阶段,又有多模块** | 当前 MVP 这种大型项目 |

**中型项目的选择逻辑**:
- 时间长、阶段明显 → 用"时间导向"(🏁 里程碑层)
- 模块多、阶段不清 → 用"模块导向"(📦 一级工作层)
- 都有 → 用"完整分级"

**硬规则**:
- **同一项目内层级必须一致** — 不能部分任务用"时间导向"、部分用"模块导向"
- **跳过的层级,整个项目都不用** — 不能部分任务有 📦、部分没有
- **自顶向下连续** — 如果有 📦,就必须有 🎯(📦 不能直接挂在无 🎯 的顶层)

skill 封装后,新建项目表单会提供"**选骨架**"下拉,按用户选择生成对应的层级框架。

## 与原飞书多维表格"级别"字段的映射

原项目的"级别"选项是:
- 🏁 阶段汇总 → 映射到新的 🏁 里程碑
- 🎯 里程碑 → 如果用于标记 Gate 节点,合并入 🏁 里程碑
- 📌 具体任务 → 映射到 📋 二级工作
- 📚 历史版本 → 映射到 **📋 二级工作**(**不是** 📌 三级工作 —— 历史版本本身就是已完成的叶子交付,符合二级标准)

## 非叶子任务的完成一致性(字段驱动约束)

**设计原则**:非叶子任务(🎯/🏁/📦)的"状态=已完成"**必须**要求所有子任务都已完成。

**实施**(飞书 Base 不支持硬强制,用"字段驱动 + 监控视图"软强制):

- 任务表上的 3 个字段:
  - `子任务总数`(lookup,自动计算)
  - `子任务已完成数`(lookup + where 状态=已完成,自动计算)
  - `完成是否合规`(formula):
    - 叶子任务 → "N/A"
    - 非叶子 + 状态=已完成 + 子任务全部完成 → "✅ 合规"
    - 非叶子 + 状态=已完成 + 有子任务未完成 → "❌ 违规"
    - 其他情况 → "—"
- 任务表上的视图 `状态异常监控`:筛选 `完成是否合规 = "❌ 违规"`

**使用规范**:
- 成员不能自由将非叶子的状态设为"已完成",必须所有子任务先完成
- 项目经理每日扫 `状态异常监控` 视图,违规即要求修改
- 下一步可加 workflow:监听状态变更,自动给修改人发提醒

## 验证标准

**好的层级设计**满足:
- 从最顶端的目标,能在视图里一路展开到最底层叶子
- 每个叶子任务都有唯一负责人和明确交付物
- WBS 编号(1.1.1.2 格式)能表达完整路径
- 一周例会看里程碑级别即可,不需要下钻到每个叶子

## 常见误区

| 误区 | 纠正 |
|---|---|
| "里程碑是时点" vs "里程碑是阶段" | 根据团队约定统一,本方案用阶段语义 |
| "所有东西都要 WBS 编号" | 对,但编号可以不连续(1.1 下面可以只有 1.1.3,删除了原 1.1.1 / 1.1.2) |
| "层级越深管理越细致" | 错,过深导致更新成本高、状态漂移 |
| "每个里程碑必须有一级工作" | 错,小里程碑可以直接挂二级工作 |
