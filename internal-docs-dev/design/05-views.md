## 任务表视图

### 核心视图:全部任务(树状浏览)

**最重要的视图**,所有日常操作默认从这里开始。

- **类型**:grid
- **分组**:仅按「所属项目」,**不按任务级别分**
- **排序**:WBS 编号升序
- **字段顺序**(从左到右):
  ```
  任务编号 | WBS编号 | 任务级别 | 任务名称 | 负责人 | 状态 | 进度 |
  计划完成日期 | 优先级 | 模块(仅工作区) | 协作人 | 所属项目 |
  父任务 | 开始日期 | 实际完成日期 | 交付物 | 风险与阻塞 | 最近更新 |
  附件 | 任务描述 | 任务保密等级 | 更新时间 | 更新人
  ```

**设计要点**:
- 按 WBS 编号排序 = 天然得到"目标 → 里程碑 → 一级 → 二级"的线性树状顺序
- 不按级别分组 = 层级关系不被分区切断
- 重要字段前置 = 打开一眼能看到"谁在做什么、进度如何"

## 视图设计原则

1. **主视图单一入口**:所有人记住"全部任务(树状浏览)"这一个视图,其他是辅助
2. **视图筛选优于条件查询**:复杂的取数场景通过视图沉淀,不要在 record-list 里堆 query string
3. **分组最多 1 层**:飞书 Base 的多层分组会让视觉混乱;一层分组(by 项目)最合适
4. **字段顺序前重后轻**:打开视图第一眼看到的一定是最常看的字段

## Dashboard(综合看板)

**定位**:每个项目一个独立 Dashboard,作为"打开 Base 的默认页",让 PM/成员一眼看全项目进度。

**核心设计原则 · 信息密度优先**:

- 彻底不用 `statistics`(单数字)块 —— 占一整格位置却只传递 1 bit 信息
- 每个图表 block 都应承载"**多维对比 / 分布**"(2 个 group_by 或堆叠)
- Dashboard 不支持"时间线"类 block(无 gantt),时间线通过 text 引导到甘特图视图

**5 个 block 的标准骨架(方向 B · 进度全景图)**:

| # | Block | 类型 | 数据配置 | 信息密度 |
|---|---|---|---|---|
| 1 | 里程碑进度矩阵(按状态堆叠) | bar(横向) | count_all + group_by [所属里程碑, 状态],filter 任务级别=📋 二级工作 | 里程碑数 × 状态数 |
| 2 | 一级工作完成率对比(%) | column | AVG(任务完成分) + group_by 父任务(value desc),filter 任务级别=📋 二级工作 | 一级工作数 |
| 3 | 时间压力分布(按到期时段+状态) | column(堆叠) | count_all + group_by [到期时段, 状态],filter 任务级别=📋 二级工作 AND 状态≠已取消 | 5 时段 × 状态数 |
| 4 | 任务状态(按级别堆叠) | column(堆叠) | count_all + group_by [任务级别, 状态] | 4 级别 × 状态数 |
| 5 | 说明 | text | Markdown 操作指引 + 图表说明(引导打开视图) | 说明 |

**排版结论**:
- 说明 text 必须最后创建,且不要再跑 arrange —— `+dashboard-arrange` 会把 text 置顶
- 推荐流程:create 4 图 → arrange → create text(text 默认追加到末尾)

**依赖的辅助字段**(任务表 formula):
- `任务完成分`:IF(状态=已完成, 100, 0) — block 2 AVG 聚合用
- `到期时段`:IFS 分档 已过期/本周到期/下周到期/后续/无 deadline — block 3 X 轴分组用
- `所属里程碑`:IFS 层级追溯到 🏁 里程碑名 — block 1 X 轴分组用

**项目隔离**:所有 block 的 `data_config.filter` 必须含 `所属项目 contains <项目名>`,让 dashboard 只聚焦当前项目。

**跨项目扩展**:新增项目时,为其单独创建 dashboard,filter 值改成对应项目名即可。Skill 封装后可模板化:"新建项目 → 克隆 dashboard 模板 → 批量替换 filter 值"。

**不能做的**:
- Dashboard block 不支持"嵌入表格视图"(无 view 类 block),想看明细列表要切到对应视图
- 不支持条件着色(如按状态染色)需通过分组实现
- 布局只能靠 `+dashboard-arrange` 自动排列,无法精确指定坐标

**已知限制 · 关键踩坑**:

1. **Filter 对 link 字段的 `contains` 操作符不生效**,会过滤出 0 条记录。实测 `contains "项目名"` 导致所有 block 显示 0。
2. **Filter `conditions` 不能为空**:`[]` 会报 `"conditions 不能为空"`。若需要"无过滤",用 `{"field_name":"任务编号","operator":"isNotEmpty"}` 做占位,等价于全量。
3. **每项目独立 dashboard 的实现策略**(已验证可行):
   - 任务表加 `所属项目名称` lookup 字段(type: lookup,从所属项目.项目名称 拉文本)
   - Dashboard block filter 用 `{所属项目名称, is, "项目名"}` —— text 字段的 `is` 稳定生效
   - 多项目时:克隆 Dashboard 模板,批量替换 filter 的 value 值即可
   - **注意**:用 link 字段直接做 filter 无效(`contains`/`is` 都测过返回 0),必须走 lookup 文本代理
4. **`block-update` 的 data_config merge** 是顶层 key merge:不传 filter 会保留旧 filter,必须显式传新 filter 才能覆盖。
5. **Text block 换行**:`\n` 会被服务端剥离,保留不下来。想要换行必须用 `\r` 或 `\r\n`(服务端保留 `\r`)。导致原样输入 markdown 的多段文本只会挤成一行 —— 构造 JSON 时用 `\r\n` 替代 `\n`。
6. **`+dashboard-arrange` 对 text 的处理**:自动把 text block 放到 **最顶部**(y=0)。用户明确要 text 在底部时,流程必须是"先 arrange 图表 → 再 create text",**不能**在 text 存在时跑 arrange。
7. **Dashboard filter 不支持 formula 字段直接做 value**:filter 的 value 必须是常量。想用动态值(如 TODAY())需在任务表加一个 formula 字段先算出分档(如 `到期时段`),再用 `is`/`contains` 筛选。

## 甘特图关键配置

甘特图最容易踩的坑:**title 字段默认指向主字段**,我们主字段是 `任务编号`(T-XXXX),**不是任务名称**。

必须配置:
- `title` = 任务名称(不是任务编号)
- `start_time` = 开始日期
- `end_time` = 计划完成日期

**可见字段建议精简为 4 列**:任务级别 / 任务名称 / 负责人 / 状态。其他信息进入记录详情页再看,不在甘特图左侧面板堆。

**排序**:WBS 编号升序。WBS 字符串本身是层级化的(1/1.1/1.1.1),升序 = 树的深度优先遍历;飞书 Gantt 自动识别 `父任务` 字段形成树状渲染,WBS 升序能让父子相邻显示,配合层级折叠效果最好。

**不要用的排序**:
- 按 `任务级别` 排序 → 所有目标聚在一起、所有里程碑聚在一起,父子被打散,折叠失效
- 按 `计划完成日期` 排序 → 早期和后期任务打散,跟层级结构冲突

**层级折叠(平台限制,无法实现)**:

⚠️ 飞书 Gantt 的左侧面板是**扁平列表**,**不提供**"父任务 → 子任务"的行级折叠。

- **不按任务级别分组绕开** —— 那会打散父子相邻,破坏 WBS 树状顺序
- **保留父子相邻 + 单层项目分组 + WBS 升序** 是当前最优方案
- **如果想聚焦高层视图**(只看 🎯/🏁/📦),切换到 `里程碑看板` 视图(Grid 类型,已过滤任务级别)

**权衡取舍**:
- Gantt 视图的价值 = 时间线 + 条形图,不是层级折叠
- 层级折叠的场景应该在 Grid 类视图解决(或手动切换 `里程碑看板` 视图)

**UI 手动步骤**:条形图颜色默认单色,若要按状态着色需在飞书 UI 里点 "条形图样式 → 按字段着色 → 状态" —— CLI 不支持此配置。

## CLI 配置视图的注意事项

**动态"当前用户"过滤** — CLI 不支持,只能在 UI 里手动加:

```
筛选 → 负责人 → contains → 当前用户
```

因此:`我的任务` / `我负责的项目` 这两个视图 CLI 建出来是空过滤的,**必须提醒用户首次使用时手动加一次**,保存后生效。

## UI 手动配置清单(首次使用必做)

> 这些操作 CLI 无法完成,需要 UI 里点一次,保存后永久生效。

**必做**(影响功能):
1. `我的任务` 视图:添加筛选 `负责人 包含 当前用户` 或 `协作人 包含 当前用户`(两条用"或"连接)
2. `我负责的项目` 视图:添加筛选 `项目经理 包含 当前用户`

**建议做**(体验优化):
3. `甘特图` 视图:工具栏 → 条形图样式 → 按字段着色 → 选"状态"
4. 所有 Grid 视图:把"任务编号"列宽拖到最窄(系统强制主字段首列,CLI 无法隐藏)

### 根本原因

飞书 Base 的"当前用户" / "条形图颜色"属于**动态渲染配置**,不是静态字段值,CLI 的 JSON payload 层面无法表达这类意图。只能由 UI 交互保存后,以配置形式嵌入视图定义,服务端在渲染时动态解析。

Skill 封装后,应该在**新建工作区的后处理步骤**里,显式提示用户完成这几项 UI 配置;或者在 skill 文档里用截图+动图引导。

## 视图跟随工作区克隆

当用户从模板克隆新工作区时,所有视图会自动复制。但**表单视图的题目引导说明 / 视图排序中的字段 ID** 需要重新测试是否生效。

## 视图与权限的关系

视图**不提供安全隔离** — 视图筛选只是 UI 便利,不拦截数据访问。真正的权限靠**高级权限角色配置**。不要把"我的任务"视图当作"成员只能看自己的"的实现手段。
