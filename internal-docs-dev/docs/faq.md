# 常见问题

## 使用场景

### Q1: 我在「我的任务」视图里看不到我被指派的任务

**原因**:CLI 创建视图时不支持"动态当前用户"筛选。

**解决**:
1. 打开任务表,切到「我的任务」视图
2. 点视图工具栏的"筛选"按钮
3. 添加条件:`负责人` contains 选 "当前用户"
4. 保存

"我负责的项目"视图同理。

---

### Q2: 父子任务展开没反应

**原因**:你可能在看板/甘特图视图上,这类视图不展示父子关系。

**解决**:切到「全部任务(树状浏览)」视图,按 WBS 编号排序可以看到树形层级。

---

### Q3: 想新增一个项目但不知道放哪个工作区

**原则**:同业务方向放同一个工作区。

- 如果是 AI 产品类 → 放 `项目管理中心·AI产品开发`
- 如果是新方向 → 联系超管(张诏瑜)评估是否新建工作区

---

### Q3.5: 任务编号 T-XXXX 为什么不连续,T-0001 到 T-0016 去哪了

**这是飞书 auto_number 字段的标准行为,不是 bug**:

- 每新建一条记录 → 计数器 +1 分配给它
- 删除记录 → **编号永久作废,计数器不回退**
- 目的:保证编号**唯一且不重复**,即使某条记录删除后也不会被重新使用

**所以为什么从 T-0017 开始**:
- 最初迁移测试建了 16 条示例数据(T-0001 ~ T-0016),后来删除了
- 实际的 MVP 项目 82 条记录从 T-0017 起编号

**缺口 T-0021 是**方案 B 重构时删除"历史归档"一级工作的位置。

**怎么正确理解编号**:
- 任务编号 T-XXXX = **系统唯一标识**,用于 URL、权限日志、跨系统对齐
- **WBS 编号**(1 / 1.1 / 1.1.2)= **业务层级标识**,用于排序、汇报、阅读
- 两者分工明确,不要指望 T-XXXX 连续

**如果真觉得不爽**:只能通过重建任务表来重置计数器(代价大,不建议)。

---

### Q4: 保密等级选错了怎么办

**无副作用,随时改**:

1. 进入项目主表,改「保密等级」字段值
2. 任务表的 `项目保密等级` lookup 字段自动同步
3. 权限立即生效

---

### Q5: 我不想做 PM 了,怎么转交

1. 在项目主表里,把「项目经理」字段改成新的人
2. 权限自动切换(字段驱动的好处)
3. 新 PM 如果还没加入"项目经理"角色,由超管添加

---

### Q5.5: 哪些表我能改,哪些只能看?

- **项目主表**:PM 可改自己负责的,成员只读
- **任务表**:成员可改"自己负责/协作"的任务(状态/进度/最近更新等),其他任务只读
- **速查卡·字段分级权限**:所有人只读,只有超管可改

权限由高级权限角色配置自动控制,不用自己想"能不能改",系统会拦下不让改的场景。

### Q6: 非项目成员能看到我的项目吗

取决于保密等级:
- **L1 公开**:可以看到项目主表记录 + 所有任务(甚至不是成员)
- **L2 常规**:只有项目成员能看到
- **L3/L4**:只有项目经理 + 负责人/协作人能看到对应任务

## 技术实现

### Q7: lookup 字段可以作为权限条件吗

**是的,已验证**。创建角色后,通过 `role-get` 查看会返回:

```json
{
  "field_type": "Lookup",
  "reference_type": "SingleSelect",
  "is_invalid": false
}
```

关键是 `is_invalid: false`,说明筛选生效。

---

### Q8: 模板改了,现有工作区自动同步吗

**不会**。模板和工作区是"一次性克隆"关系,**后续互不影响**。

如果模板做了改进,需要:
1. 把改进手动应用到每个工作区(脚本化),或者
2. 新建工作区从最新模板开始

**这是有意的设计**:避免模板改动影响生产数据。

---

### Q9: 如何备份

飞书 Base 会自动保存历史版本,一般不需要主动备份。

如果要离线备份,可以用:
```bash
lark-cli base +record-list --base-token <token> --table-id <tbl> --limit 200 > backup.json
```
每张表导出一次。

---

### Q10: 跨多个工作区,怎么聚合"我的任务"

**目前没有原生方案**。

临时方案:
- 在每个工作区都手动看"我的任务"视图
- 或用 API 跨 Base 聚合(需要脚本)

**长期方案**(待讨论):在模板上加一张"跨工作区同步表",通过 workflow 自动聚合。

### Q12: Skill 和飞书 Base 的关系

- **Skill**:运行在用户本地的 AI Agent 上,通过 lark-cli 调用飞书 API
- **飞书 Base**:数据和权限都在飞书服务端,Skill 不存数据
- **关系**:Skill 是"搭积木的工具 + 规范约束",Base 是"积木成品"
