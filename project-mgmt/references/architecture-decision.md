# 架构决策树 · 团队规模 × 项目并发

> wizard 阶段 D 推断架构时读这个文件。

## 三种架构方案对比

| 方案代号 | 用户友好名 | 结构 |
|---|---|---|
| **A** | 轻量共享 | 1 个 Base,1 张共享任务表 |
| **B1** | 多任务表 | 1 个 Base,每项目独立任务表 + 独立 dashboard |
| **B2** | 多空间分离 | 每项目 1 个独立 Base,Master Base 做总览 |

## 决策矩阵

| 团队 \ 项目 | 1-3 个 | 4-10 个 | 10+ 个 |
|---|---|---|---|
| <5 人 | **A** | B1 | B2(+警告) |
| 5-15 人 | B1 | **B1**(主流) | B2 |
| 15+ 人 | B1 | B2 | **B2** |

**粗体 = 强推荐**;非粗体 = 可推荐但有更优选项。

## 三方案的关键差异

### A · 轻量共享(单表)

**优点**:
- PM 跨项目天然总览(同张表分组按项目即可)
- 新项目零成本(项目主表加 1 行)
- Schema 演进只改 1 张表

**缺点**:
- 项目数 5+ 时视觉拥挤
- 成员可能看到其他项目(靠权限行级过滤,但视觉上仍乱)
- 大表性能可能变慢(1000+ 行)

**适合**:1-3 个临时项目,小团队 / 个人玩耍。

### B1 · 多任务表(同 Base)

**优点**:
- 物理隔离 + PM 不切 Base(在一个空间内切表)
- 每项目独立 dashboard
- 统一权限(Base 级)
- 新项目克隆 1 张表 + 1 个 dashboard,skill 自动化

**缺点**:
- Schema 演进要在 N 张表同步(skill 提供 `pm add-field --to all-tasks` 批量命令)
- 飞书 Base 单 Base 表数量上限(免费 ~20,高级 ~50)→ 10 项目左右上限

**适合**:**3-10 个并行项目,中型团队**。绝大多数公司中流砥柱。

### B2 · 多空间分离

**优点**:
- 每项目完全隔离(Base 级)
- 项目间权限/保密互不影响
- 表数量上限不是问题

**缺点**:
- PM 多项目要切 Base(浏览器多 tab)
- 跨项目总览没有原生方案(需要 Master Base + API 同步)
- 新项目加入需要克隆 1 个 Base(更重,但 skill 仍可自动化)
- Schema 演进要在多个 Base 同步

**适合**:10+ 个项目,跨团队联合,严格保密分级,或单个项目特别大。

## 路由说明

skill 阶段 D 屏 3 应:

1. **告诉用户推荐方案**(用友好名,不用 A/B1/B2)
2. **说明 3 个核心理由**(适合哪种工作方式)
3. **可让用户选"看其他方案"**显示完整对比
4. **强烈不推荐时给警告**(如 <5 人选 B2)

## 🚨 Q5 = 10+ 时的安全审核(避免误推 B2)

用户在 Q5 答「10+ 个项目」时,skill **必须先做一次澄清**,再走架构决策矩阵。原因:

- 用户经常把"我团队总共管的项目数"和"放在这一个空间里的项目数"混淆
- 若答错(报上跨方向的总数),wizard 会推 B2 多空间架构 → 装更慢、PM 切 Base 累、跨项目总览困难
- 真实场景里,**5-15 人团队同方向项目数超过 10 的极少**;往往是用户混淆了 scope

### 澄清交互(skill 在跑决策矩阵前必问)

```
你选了「10+ 个项目」 — 我想再确认下:

  这 10+ 个项目是不是都属于**同一个业务方向**?
  (比如都是产品研发,或都是客户交付)

  [1] 是 — 都是同方向,需要在一个空间里统一管
       └ 走原决策矩阵 → 多半推 B2 多空间架构
  [2] 否 — 跨多个方向,可以拆
       └ 引导用户:**先建当前最主要方向的空间**,
          只填那个方向的项目数(很可能 4-10),
          其他方向回主菜单单独建

▸ 选 1-2:
```

- 选 [1] → 按原矩阵推 B2(明确告知"这是多空间架构,装完后每项目一个 Base")
- 选 [2] → skill 跳回 Q5 让用户重答**主方向**的项目数(此时通常落到 [2] 4-10,推 B1)

### 这条审核不能跳过

- 即使在 wizard 调试 / 自动化 / fast-mode 下,Q5 = 10+ 都要走这一问
- 这是设计原则:**当用户的回答可能源自意图歧义时,skill 多问一次比让用户重装空间更便宜**

## 边界规则

- 用户选 **A 方案** → 后续要展示一句:**"将来项目多了可平滑升级到 B1。skill 提供 `pm migrate-to-multi-tables` 命令"**
- 用户选 **B2 方案** → 警告"配置稍复杂,需 1.5 倍时间(~7 分钟)"
- **架构选择写入 `wizard.state.json` `_inferred_arch` 字段**,后续场景 B 加项目时按这个值走相应路径

## 何时建议用户重新选择

- 用户跑了一段时间后,如果 B1 顶到表数上限(10+ 项目),提示迁移到 B2
- 项目数从 1-3 涨到 5+ 时,提示从 A 升 B1
- 这些迁移由用户主动跑 `pm migrate-*` 命令触发,**不要自动迁移**
