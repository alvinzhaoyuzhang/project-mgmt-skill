# 从飞书云文档协助新建/更新

> 当用户给出一个飞书 docx 链接(会议纪要 / 周报 / 工作计划 / PRD 等),要求从中**批量建项目结构**或**批量更新任务完成情况**时读这个文件。
>
> 这是场景 B(加项目)和场景 C(任务操作)的 doc-driven 变体,**不开新场景**,只是把"问 wizard 5 题"换成"读 doc 提取信息"。

## 适用场景识别

用户输入含**两类**关键信号:

| 信号 | 含义 | 路由 |
|---|---|---|
| docx URL + "新建项目"、"加项目"、"建任务" | E1 · 从文档建项目 | 走本文档 § E1 |
| docx URL + "更新进度"、"批量同步"、"周报"、"会议纪要" | E2 · 从文档更新任务 | 走本文档 § E2 |

如果用户给了 docx 但意图不明,先**问一句**:"你想让我从这个文档**新建项目结构**,还是**更新现有任务的状态**?"

## 跨 skill 协作:委托给 lark-doc

读飞书 docx 是 `lark-cli docs +fetch` 的能力域,**优先委托 lark-doc skill** 完成文档读取。

```bash
# 提取 docx 内容(让 lark-doc skill 处理或直接调命令)
lark-cli docs +fetch --doc-token <docx_token>
```

`<docx_token>` 从 URL 提取,格式如 `https://xxx.feishu.cn/docx/<docx_token>`(`docx` 前缀的部分)。

读到的内容传给本 skill 的解析逻辑(后续步骤),**不要**让本 skill 自己 fetch。

---

## E1 · 从文档建项目结构

### 步骤 1 · 拿到 docx token

用户输入 URL 后,提取 token。如果 URL 形如 `/wiki/<wiki_token>`,要先用 `lark-cli wiki spaces get_node` 解析为真实 obj_token。

### 步骤 2 · 委托读 doc

```bash
lark-cli docs +fetch --doc-token <token>
```

返回 markdown 形式的内容(含标题、列表、表格)。

### 步骤 3 · LLM 解析提取

从文档识别这些结构(以会议纪要为例):

| 文档元素 | 映射到 |
|---|---|
| 项目名(标题或开头明示) | 项目主表"项目名称" |
| 大段落标题(如"里程碑 1") | 🏁 里程碑 |
| 小段落标题或粗体(如"前端开发") | 📦 一级工作 |
| 列表项 / 表格行 | 📋 二级工作 |
| @某人 / "由 X 负责" | 任务"负责人" |
| "X 月 Y 日前完成" | 任务"计划完成日期" |
| "P0" / "紧急" / "高优" | 任务"优先级" |

工作计划 / PRD 类文档结构更规整,可能有现成的"任务编号 / 负责人 / 截止日"表格,直接 1:1 映射。

### 步骤 4 · Preview 给用户

```
我从文档识别到:

项目:Q3 战役
├── 🏁 里程碑 1:启动 (5/20 前)
│   ├── 📋 调研竞品(王芳)
│   ├── 📋 立项评审(@周经理)
│   └── 📋 资源申请(孙磊)
├── 🏁 里程碑 2:开发(6/30 前)
│   ├── 📋 接口设计(王芳,P0)
│   └── ... (5 个任务)
└── 🏁 里程碑 3:发布(7/15 前)
    └── ... (3 个任务)

合计:1 项目 + 3 里程碑 + 8 任务
负责人映射:王芳 / 周经理 / 孙磊(都需要验证飞书账号)

请审核:
  [1] 全部接受,直接执行
  [2] 我要逐条修改(进入对话调整)
  [3] 重新解析(可能你的文档我读得不准)
  [4] 取消
```

### 步骤 5 · 用户调整后执行

- 调用 `scripts/new-project.sh` 建项目(项目主表加行 + 克隆任务表 + 建 dashboard)
- 然后批量 `+record-batch-create` 加 N 个任务
  - 父子关系:里程碑→工作包→叶子任务,逐层创建并记录 record_id
  - 后建的任务的"父任务"link 指向先建的父记录 record_id

### 步骤 6 · 关联原文档

在项目主表对应行的"备注"或"原始文档"字段填上 docx URL,方便日后追溯出处。

---

## E2 · 从文档批量更新任务

### 步骤 1-2 · 同 E1(拿 token + fetch)

### 步骤 3 · LLM 识别任务更新片段

从周报 / 站会记录里识别这些语义(典型句式):

| 句式 | 映射 |
|---|---|
| "X 完成了 / 已交付 / 上线" | 状态=已完成,进度=100 |
| "X 进度 50% / 一半 / 大半" | 进度=数字 |
| "X 卡住了 / 等 Y / 阻塞" | 状态=阻塞,风险与阻塞=原文 |
| "X 延期到 Y 月" | 计划完成日期 改为 Y |
| "X 转给 Z 了" | 负责人改为 Z |

### 步骤 4 · 飞书内匹配任务

每个识别出的"任务名称"片段,跑 `+record-search` 找飞书空间内对应任务:

```bash
lark-cli base +record-search --base-token <token> --table-id <tid> \
  --json '{"keyword":"<任务名片段>","search_fields":["任务名称"]}'
```

分类:
- **唯一匹配** → 直接 propose update
- **多匹配** → 让用户选哪个(列出 [1] [2] [3] 候选)
- **0 匹配** → 标记为"未识别",后置让用户处理

### 步骤 5 · Preview · 关键

```
从周报识别到 7 处任务更新,匹配如下:

✅ 4 处自动匹配
  • "实现登录接口" → T-0042(王芳)→ 改进度 60→100,状态→已完成
  • "支付回调" → T-0058(李雷)→ 改进度→50
  • ...

⚠️  2 处多匹配,请选
  • "测试" 在飞书内匹配到 3 条:[1] T-0061 单测 [2] T-0072 集成测试 [3] T-0089 冒烟测试
    → 你要更新哪个?或全部?

❌ 1 处未在飞书找到
  • "X 模块设计稿" → 没找到。是不是任务名跟文档里写法不同?

请审核每条,确认后批量执行。
```

### 步骤 6 · 批量执行

用 `+record-batch-update`:

```bash
lark-cli base +record-batch-update --base-token <token> --table-id <tid> \
  --json '{
    "record_id_list":["<rid1>","<rid2>",...],
    "patch":{"状态":"已完成"}
  }'
```

注意:**不同记录的 patch 不同**时,要分批跑(每批一种 patch)。

## 关键约束(对 skill 的要求)

1. **永远 Preview 后才执行**:LLM 解析有可能误识别,必须让用户审核
2. **匹配不上的不要瞎建**:E2 流程中 0 匹配的任务**不要默默跳过**,要列出来让用户决定(是否新建?是否改写法重搜?)
3. **进度数字看大概**:文档里说"一半" → 50,"大半" → 75,"快做完" → 90;但这种模糊推理要在 preview 里标 "(估计)",让用户改
4. **保留原文上下文**:"最近更新"字段写入时,**带上文档原文片段**(如 "[来自 2026-04-26 周报] X 任务联调完成,正在补 case")
5. **不直接 @ 通知人**:E2 只更新任务字段,**不主动**发飞书消息(改了"负责人"飞书会自动通知,这是平台行为不是 skill 主动)

## 边界场景

| 情况 | 处理 |
|---|---|
| docx 是 PDF / 图片 | 提示用户:目前只支持飞书原生 docx,PDF 让用户先转 docx 或手动拷贝内容 |
| docx 太长(>5000 字) | 提示:"文档很长,我先列我识别到的任务清单(估计 N 条),你扫一眼对不对再继续" |
| docx 中的人名是昵称 / 简称 | 用 `lark-cli contact user.search` 搜可能的全名;搜不到让用户给 open_id 或邮箱 |
| docx 是 wiki 链接 | 先 `lark-cli wiki spaces get_node` 解析 → 拿 obj_token → 再 `+fetch` |
| 用户没说哪个空间 | 同其他场景:列出 registered_bases.json 让用户选 |
| 用户给了 docx 但还想加 wizard 默认值 | 优先用 doc 内容,缺的字段(如保密等级)再问 |

## 不支持的事(明确告知用户)

- ❌ 不支持图片/截图 OCR(让用户先转文字)
- ❌ 不支持 docx 实时同步(每次都是手动触发)
- ❌ 不支持复杂表格内嵌的多层结构推断(简单表格 OK)
