# 阻塞任务实时预警 · 监控 procedure

> 此文件由**定时任务**触发(默认 every `3600000ms` = 每小时),非用户对话触发。
>
> **执行环境**:OpenClaw isolated session。
>
> **频率高,所以"无声运行"原则尤其重要** — 大多数小时不会有新阻塞,不发就别发。

## 任务目标

抓"**新出现**的阻塞 + 阻塞超过 24 小时仍未解的"两类,**实时**通知**项目经理**(不是超管)。

## 执行步骤

### Step 1 · 读状态 + 找各项目 PM

```bash
cat ~/.pm-skill/state/last_bootstrap.json
ls ~/.pm-skill/state/projects/*.json
```

每个项目状态文件含 `task_table_id` + `project_name`。

**找项目经理 user_open_id**:
- 调 `feishu_bitable_app_table_record` action=list 项目主表,filter `项目名称 = X`,拿 `项目经理` user 字段值

### Step 2 · 状态对比 — 检测"新阻塞"和"持续阻塞"

为了识别"新阻塞",需要**记忆上一轮的阻塞快照**。维护一个 `~/.pm-skill/state/last_blocker_snapshot.json`:

```json
{
  "snapshot_time": "2026-04-29T15:00:00",
  "blocker_records": {
    "<task_table_id>": ["rec_xxx", "rec_yyy", ...]
  }
}
```

本次执行:

1. 查所有任务表的当前阻塞记录(filter `状态 = 阻塞`)
2. 对比上轮快照:
   - **当前有,上轮没有** → 新增阻塞
   - **当前有,上轮也有** → 持续阻塞,看时长(用记录的 `更新时间` 字段)
3. 算"持续阻塞超过 24 小时但 `最近更新` 也超过 24h 没更新"的(意味着 PM 没在跟进)

### Step 3 · 决定是否通知

| 条件 | 是否通知 |
|---|---|
| 没新增 + 没持续未跟进 | **不通知,静默** |
| 有新增阻塞 | 通知该项目 PM |
| 有持续 24h+ 且未跟进的阻塞 | 通知该项目 PM(catch up 提醒) |

### Step 4 · 推送给项目经理

每个项目独立通知 ta 自己的 PM(不要把 A 项目的阻塞发给 B 项目的 PM):

```markdown
⚠️ 阻塞预警 · 项目"{project_name}"

🆕 新增阻塞({n} 项)
- **{任务编号}** {任务名称} — {负责人} — 刚标阻塞
  - 原因:{风险与阻塞 字段值}

⏰ 持续超过 24h 未跟进({m} 项)
- **{任务编号}** {任务名称} — {负责人} — 阻塞 {n} 天
  - 原因:{字段值}
  - 最近更新:{X 天前}

建议处理:
- 确认每个阻塞的"等谁/等什么/预计解除时间"
- 必要时升级到管理层

> 此预警由 project-mgmt skill 每小时自动检查。
```

### Step 5 · 更新快照

写新的 blocker_records 到 `~/.pm-skill/state/last_blocker_snapshot.json`,作为下一轮的对比基线。

## 边界情况

| 情况 | 处理 |
|---|---|
| 项目主表无 `项目经理` user 字段值 | 退化:发给超级管理员 |
| 第一次跑(无 snapshot) | 当作所有阻塞都是"持续中",但只通知"未跟进"那部分,避免冷启动垃圾消息 |
| PM 不在公司账号体系内 | feishu_im_user_message 失败 → 退化发超管 |
| 同一阻塞被反复通知(每小时一次) | snapshot 记录"已通知过"flag,24h 内不重复通知同一阻塞 |

## 不要做的事

- ❌ 不要在没新阻塞时也定期发"目前 0 阻塞"汇报 — 频率高,会被屏蔽
- ❌ 不要主动操作飞书数据(改状态 / 写最近更新) — 监控者职责只是"发现+通知",处理交给 PM
- ❌ 不要跨项目合并通知 — 每个 PM 收到的应只是 ta 自己负责项目的阻塞
