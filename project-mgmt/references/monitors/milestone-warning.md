# 里程碑到期预警 · 监控 procedure

> 此文件由**定时任务**触发(默认 cron `0 9 * * *`,每天 9:00),非用户对话触发。
>
> **执行环境**:OpenClaw isolated session。

## 任务目标

每天检查未来 7 天内到期的里程碑,**重点是看里程碑下叶子任务的完成情况**——如果里程碑临近 deadline 但叶子任务进度不足,主动给项目经理发预警。

里程碑是项目控制点,跟普通叶子任务的预警逻辑不同 — 里程碑达成要看下属任务全部交付,所以预警要算"里程碑健康度"。

## 执行步骤

### Step 1 · 找未来 7 天到期的里程碑

```bash
cat ~/.pm-skill/state/last_bootstrap.json   # 拿 base_token / 项目主表 id
ls ~/.pm-skill/state/projects/*.json        # 所有项目
```

对每个项目的任务表,调 `feishu_bitable_app_table_record` action=list:

```yaml
filter:
  conjunction: and
  conditions:
    - field_name: "任务级别"
      operator: "is"
      value: "🏁 里程碑"
    - field_name: "状态"
      operator: "isNot"
      value: "已完成"
    - field_name: "状态"
      operator: "isNot"
      value: "已取消"
    - field_name: "计划完成日期"
      operator: "lessOrEqual"
      value: "<今天 + 7 天>"   # 7 天内到期
    - field_name: "计划完成日期"
      operator: "greaterOrEqual"
      value: "<今天>"
```

### Step 2 · 对每个待到期里程碑,算"健康度"

对每个匹配的里程碑:
1. 拿它的 record_id
2. 列出所有以该 record_id 为父任务的叶子任务(直接子任务 + 子任务的子任务,深度可能 2-3 层)
3. 算指标:
   - **完成率**:已完成数 / 总叶子数
   - **预测达成日期**:剩余任务总剩余时长(假设线性)→ 看是否 ≤ 里程碑计划完成日期
   - **阻塞下属**:多少个叶子是 阻塞 状态

### Step 3 · 风险分级

| 级别 | 条件 |
|---|---|
| 🔴 **高危** | 里程碑剩余天数 ≤ 3 且 (完成率 < 70% 或 阻塞下属 ≥ 1) |
| 🟡 **关注** | 里程碑剩余天数 ≤ 7 且 完成率 < 50% |
| 🟢 健康 | 其他 |

只对 🔴 / 🟡 发预警,🟢 静默。

### Step 4 · 推送

每项目找 PM(同 blocker-alert procedure),发:

```markdown
🎯 里程碑预警 · 项目"{project_name}"

🔴 高危({n} 项)
- **{里程碑名}** — 还剩 {N} 天 — 完成率 {X}% — {阻塞数} 个下属阻塞
  - 下属任务总数 / 已完成 / 进行中 / 阻塞 / 未开始
  - 关键阻塞(若有):{任务编号} {任务名称} — {负责人}

🟡 关注({m} 项)
- **{里程碑名}** — 还剩 {N} 天 — 完成率 {X}%
  - ...

建议:
- 高危里程碑:今天就拉相关人对齐,看是否需要砍范围或延期
- 关注里程碑:确认下属任务都在按时推进,提前发现风险

> 此预警由 project-mgmt skill 每天 9:00 自动检查。
```

## 边界情况

| 情况 | 处理 |
|---|---|
| 里程碑无下属任务 | 风险等级 = 🟡(因为没数据可看,提示用户补充),提示"该里程碑下没有挂任何叶子任务,要补吗?" |
| 里程碑下叶子是阶梯式延伸(📦 → 📋) | 递归收集所有叶子,不只看直接子节点 |
| 用户已把里程碑标 已完成 | filter 已排除,不预警 |
| 项目状态=已结案 | 该项目下所有里程碑都跳过 |

## 不要做的事

- ❌ 不要给"建议把里程碑延期"的建议 — 决策应由用户做
- ❌ 不要直接修改飞书数据(标状态 / 改日期)— 监控者只汇报
- ❌ 不要跨项目合并(同 blocker-alert)
