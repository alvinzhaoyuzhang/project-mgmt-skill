# 周报自动生成 · 监控 procedure

> 此文件由**定时任务**触发(默认 cron `0 10 * * 5`,周五 10:00),非用户对话触发。
>
> **执行环境**:OpenClaw isolated session。

## 任务目标

每周五上午 10:00,**自动**给所有活跃项目跑一份周报,推送到超级管理员。让用户开周会前已经有现成材料。

## 执行步骤

### Step 1 · 读状态文件,枚举所有项目

```bash
cat ~/.pm-skill/state/last_bootstrap.json    # 拿 base_token + 项目主表 id
ls ~/.pm-skill/state/projects/*.json          # 列出所有已注册项目
```

对每个项目,从 `projects/<项目名>.json` 拿 `task_table_id` + `project_name` + `dashboard_id`。

### Step 2 · 对每个项目,跑周报 SQL-style 聚合

调 `feishu_bitable_app_table_record` action=list 拿任务表数据。

**本周时间窗**:周一 00:00 → 周日 23:59(基于触发那天的 ISO 周)。

**计算 5 个 section**:

#### Section 1 · 进度数字
```
项目 [X]
  累计完成率: 35%(35/65 已完成 / 总叶子)
  本周完成: 5 条
  进行中: 12
  阻塞: 1
  延期: 1
```

#### Section 2 · 本周亮点
- 本周达成的里程碑:`实际完成日期 ∈ [本周一, 本周日] 且 任务级别 = 🏁 里程碑` 的任务名
- 各成员本周完成的叶子任务

#### Section 3 · 风险与阻塞
- 当前阻塞:`状态 = 阻塞` 的任务(用 `风险与阻塞` 字段内容做摘要)
- Top 3 延期:`计划完成日期 < 今天 且 状态 ∉ (已完成,已取消)`,按延期天数排序

#### Section 4 · 下周计划
- 下周一到周日到期的任务:`计划完成日期 ∈ [下周一, 下周日]`,按日期升序

#### Section 5 · 成员进展
- `最近更新` 字段非空且本周更新过的任务,按负责人聚合

### Step 3 · 组装 markdown

每个项目 1 段,所有项目拼成一份完整周报。

```markdown
# 项目周报 · 第 {ISO 周号} 周({日期范围})

## 📦 项目 1:用户标签 v2

### 📊 进度数字
- 累计完成率:**35%** (35 / 65,已取消 0)
- 本周完成:5 条 / 进行中:12 / 阻塞:1 / 延期:1

### 🎯 本周亮点
- ✅ 里程碑达成:Beta 发布前期(2026-04-30 完成)
- ✅ 张三完成:用户标签服务开发
- ✅ 李四完成:标签管理界面 UI

- 🟡 个人主页改版(赵六)— 延期 2 天 — 前端工程师调走

### 🗓 下周计划(W19)
- 标签管理界面联调(李四,周二到期)
- 个人主页改版(赵六,周四到期)

- 张三:完成标签服务,下周开始接入推荐
- 李四:UI 80%,周末完工详情页
- ...

## 📦 项目 2:其他项目

(同样 5 section)

> 此周报由 project-mgmt skill 每周五 10:00 自动生成。
> 完整数据可在飞书 base 查看:https://...
```

### Step 4 · 推送

调 `feishu_im_user_message` 给**超级管理员**,msg_type=post(富文本)。

也可选:用 `feishu_create_doc` 把这份周报转成飞书 doc(若 markdown > 4000 字符更适合)。

## 边界情况

| 情况 | 处理 |
|---|---|
| 没有活跃项目 | 发"本周无活跃项目"简短通知,而不是不发(用户至少知道 skill 在运行) |
| 项目 X 任务表为空 | 跳过该项目 section |
| 时间字段值为 0 / null | 当作"无 deadline",归到下周计划"无 deadline 部分" |
| 单项目任务 > 500 | 取 page_size=500;超过 500 在报告末尾标注"任务数过多,只统计前 500 条" |

## 不要做的事

- ❌ 不要预测"项目能否按时完成" — 数据信号有限,容易误导
- ❌ 不要给主观评论(如"项目状况良好") — 只列数据
- ❌ 不要在周报里推销其他 skill 功能
