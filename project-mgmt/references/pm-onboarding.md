# PM 进项目第一天 · skill 主动 brief 流程

> **何时读这个文件**:用户说"我刚回项目"、"看下我的项目状况"、"X 项目什么情况"等**模糊的"全景式"诉求**,skill 不应反问"看什么具体维度",而是**主动给完整 brief**。

## 触发关键词识别

skill 应识别以下用户诉求并进入 onboarding 路径(不是简单查询):

- "我刚回项目"
- "看下我的项目"
- "X 项目什么情况"
- "我离开几天了,有啥变化"
- "现在啥状况"
- "给我个 overview"
- "我接手这个项目了"

## skill 应该做的(主动 brief 4 层结构)

**不要反问**"你想看什么具体维度",而是直接跑这 4 层 brief,**一次性**给用户全景:

### Layer 1 · 项目核心数字(数据层)

调 `lark-cli base +record-list` 查项目主表 + 任务表,算:

```
项目"X"健康度速览
─────────────────
  累计完成率:35% (35/65 已完成)
  里程碑达成:1/4 (Beta 发布尚在进行中)
  当前活跃叶子任务:18 个
  阻塞任务:2 个 ⚠️
  延期任务:1 个 ⏰
  P0 任务剩余:1 个(高危)
```

### Layer 2 · 关键事件(发生了什么)

跑 `lark-cli base +record-list` 加 filter:

```
最近 7 天的关键事件
─────────────────
  ✅ 完成里程碑:需求冻结(2026-04-15)
  ✅ 完成 5 个叶子任务(用户访谈、PRD 撰写、PRD 评审...)
  🔴 新增阻塞:推荐引擎接入(等数据团队,3 天)
  🟡 新增延期:个人主页改版(前端工程师调走)
  ➕ 加了 2 个新工作包(测试相关)
```

判定逻辑:
- **完成里程碑**:`任务级别=🏁 里程碑 且 实际完成日期 ∈ [今天-7, 今天]`
- **完成叶子任务**:同上但 任务级别=📋
- **新增阻塞**:`状态=阻塞 且 风险与阻塞 字段更新时间 ∈ [今天-7, 今天]` (近似:用 `更新时间` 字段)
- **新增延期**:`状态=延期 且 更新时间 ∈ [今天-7, 今天]`

### Layer 3 · 现在最该处理的(行动层)

```
3 件事建议优先看
─────────────────
  1. 🔴 推荐引擎接入(王五,P1)— 阻塞 3 天,该升级催数据团队组长了
     → 想我帮你 @ 数据组长?或转给其他人?
  2. ⏰ 集成测试(测试组,P0)— 5 天后到期,但还没开始
     → 是估时不够,还是依赖前面的工作?要不要重新评估?
  3. 🎯 GA 发布里程碑 — 22 天后到期,完成率 0%(未开始)
     → 时间表合理吗?要不要提前拆任务做准备?
```

判定逻辑:**严重风险 + 临期 P0 + 待到期里程碑** 中各取最 critical 的 1-3 件。

### Layer 4 · 接下来一句话引导

不是简单结束,而是把球传给用户:

```
你想从哪一项入手?(回我"1" / "2" / "3",或告诉我别的关注点)
```

## 不要做的事

- ❌ **不要反问**"你想看什么维度?进度?风险?成员?" — 用户问"看下项目"就是要全景
- ❌ **不要只给数字不给行动建议** — Layer 1 数字 + Layer 3 建议 是配套的,缺一不可
- ❌ **不要列所有阻塞 / 所有延期** — Layer 3 选 Top 3 就够,不要垃圾消息
- ❌ **不要用专业术语**(WBS / 叶子 / 父任务) — 用户不一定懂,用"任务"统一指代
- ❌ **不要在 Layer 4 让用户回详细方案** — 只问"哪一项",细节后续展开

## 跟其他场景的区别

| 场景 | 触发 | skill 应做 |
|---|---|---|
| **本场景(onboarding)** | "我刚回 / 看下项目状况" | 主动 4 层 brief |
| Op-PM1 跨项目状态查询 | "哪些 P0 卡住了" | 直接给 list,不做 brief |
| Op-M3 标阻塞 | "X 任务卡住了" | 标阻塞 + 追问"等什么" |
| 场景 G · 周报 | "出周报" | 跑 `weekly_report.py` |

**onboarding 是"全景"模式,其他都是"单点"模式**。区分关键看用户措辞:模糊的"看下/全貌/啥状况" → onboarding;具体的"X 任务/P0 卡住/出报告" → 其他。
