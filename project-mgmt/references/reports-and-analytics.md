# 报告与分析

> 当用户要看**项目周报、风险预警、项目复盘**时读这个文件。三个工具命令对应三种节奏:周报(每周)、风险预警(每天)、复盘(项目结束)。

## 三个核心命令

| 用户诉求 | 命令 | 频率 |
|---|---|---|
| "给我看下 X 项目这周的周报" / "本周项目进展怎么样" | `pm weekly-report --project <X>` | 每周一次 |
| "X 项目有哪些风险" / "现在哪些任务卡住了" / "看下我项目的健康度" | `pm risk-check --project <X>` | 每天一次 |
| "X 项目结束了,出个复盘报告" | `pm retrospective --project <X>` | 项目末尾 |

## 触发判断

skill 收到这类诉求时:

1. **识别诉求类型**(从用户语义,不一定要用户用准确命令名):
   - 周报关键词:**周报、本周进展、汇报、总结一周、本周完成、本周计划**
   - 风险关键词:**风险、卡住、阻塞、延期、临期、health、状态怎样**
   - 复盘关键词:**复盘、回顾、总结报告、项目结束、retrospective**

2. **识别项目名**:
   - 用户明示("Q3 上线项目")→ 直接用
   - 用户没明示但 `~/.pm-skill/registered_bases.json` 只有 1 个项目 → 默认用
   - 多个项目 → 列出来让用户选

3. **跑对应脚本**:

```bash
bash $SKILL_ROOT/scripts/weekly_report.py \
  --base-token <base-token> \
  --project-name "<项目名>"
# 或
bash $SKILL_ROOT/scripts/risk_check.py \
  --base-token <base-token> \
  --project-name "<项目名>"
# 或
bash $SKILL_ROOT/scripts/retrospective.py \
  --base-token <base-token> \
  --project-name "<项目名>"
```

输出是 markdown 字符串,**直接展示给用户**,不用再处理。

## 周报内容结构

`weekly_report.py` 输出包含:

| 章节 | 内容 |
|---|---|
| 进度数字 | 累计完成率 / 本周完成数 / 进行中 / 阻塞 / 延期 |
| 本周亮点 | 里程碑达成 + 各成员完成的任务 |
| 风险与阻塞 | 当前阻塞 + Top 3 延期任务 |
| 下周计划 | 下周一到周日的到期任务清单 |
| 成员进展 | 各成员"最近更新"字段汇总 |

## 风险预警内容

`risk_check.py` 输出三档:

| 档 | 标准 | 触发动作 |
|---|---|---|
| 🔴 严重风险 | 阻塞超 3 天 + P0/P1 延期 | PM 必须立即介入 |
| 🟡 关注项 | 一般阻塞 / 普通延期 | 知情即可 |
| ⏰ 临期 | 未来 3 天到期 | 提前关注 |
| 📊 资源紧张 | 单人进行中任务 ≥ 5 | 考虑重分配 |

阈值可通过参数调:`--severe-block-days 5` 等。

## 复盘报告内容

`retrospective.py` 输出包含:

| 章节 | 内容 |
|---|---|
| 总体数据 | 周期、任务数、完成率、按时率、阻塞次数 |
| 里程碑达成 | 各里程碑达成率表格 |
| 成员贡献 | 各成员完成任务数和占比 |
| 优先级分布 | P0/P1/P2/P3 任务比例 |
| 延期任务 | 完成但超期的任务清单 |
| 阻塞回溯 | 所有遇到过阻塞的任务和原因 |
| 经验沉淀模板 | "做得好的 / 需要改进的 / 复用建议" 三段(让团队补充) |

复盘报告**自动识别一些异常模式**:
- 按时率 < 70% → 提示"估时偏乐观"
- 阻塞率 > 20% → 提示"上下游协调有问题"
- 单人完成 > 50% → 提示"资源过度集中"

## skill 跟用户交互的样式

### 周报场景

```
用户:Q3 项目这周怎么样?
skill:[识别 = 周报] 我跑下 Q3 项目本周周报...
[bash weekly_report.py ...]
[输出整份 markdown 周报]

是否要导出 markdown 文件?或发给项目群?
```

### 风险场景

```
用户:我项目最近哪些任务有问题?
skill:[识别 = 风险预警]
[bash risk_check.py ...]
[输出三档清单]

要不要我帮你把严重风险的任务负责人都 @ 一遍?(走 Op-PM5 通知流)
```

### 复盘场景

```
用户:Q3 项目下周交付完成,你帮我做个复盘报告
skill:[识别 = 复盘]
ⓘ 我先确认:项目所有任务都已完成或确定取消了吗?
   如果还有进行中任务,复盘会包含未完成项,可能数据不全。

   [是,所有任务已结案] / [否,先等几天让任务收尾]
```

## 输出选项

skill 把 markdown 直接给用户后,可以建议:
1. **复制粘贴** — 用户自己用
2. **导出文件** — `tee /tmp/report.md`
3. **发飞书群** — 让用户提供群 ID,skill 调 `lark-cli im chat.send` 发送
4. **作为飞书 doc** — 让 lark-doc skill 帮创建一个 doc

## 不做的事

1. **不主动定时跑** — skill 是被动触发,不开自动 cron(用户想要每天报,自己设置 cron 或飞书机器人)
2. **不发邮件** — 出 markdown 即可,渠道分发用户自己定
3. **不算预测** — 不做 burndown 预估完成日期(数据信号有限,容易误导)
