# 任务变化的下游影响分析

> **何时读这个文件**:用户改任务的关键属性(状态/优先级/截止日期/所属里程碑)时,skill 不能只闷头改字段,**必须主动分析连锁影响 + 提示用户**。

## 核心理念

**单个任务不是孤立的**——它有父任务、它属于里程碑、它跟其他任务并行抢资源。改一个任务可能引发:

- 父任务 / 里程碑的进度数字变化
- 同负责人的其他任务被挤压
- 后置依赖任务的时间被推迟
- 项目整体的甘特图重新排布

skill 的价值在于:**在用户改之前,主动展示这些连锁影响,让用户做明智决策**。

## 4 类典型变化 + skill 应做的引导

### 类型 1 · 任务延期(改"计划完成日期"延后)

**触发场景**:用户说"X 任务推迟一周"、"延到下周五"、"标延期"等。

**skill 必做 4 步**:

1. **展示当前 vs 拟改值** + 延期天数
2. **找直接下游任务**(以 X 为父任务的子任务,或 X.WBS 之后顺序排的任务)
3. **找受影响的里程碑**(X 所属里程碑,看完成率变化)
4. **追问用户**:这一周延期影响的下游任务,要不要也延?

**对话样例**:
```
用户:把"标签管理界面"延到 5 月 15 日

skill:好的,我先看下连锁影响。

📋 当前:计划完成日期 2026-05-08
📋 改为:2026-05-15(延 7 天)

⚠️ 影响:
  - 这是 Beta 发布里程碑(2026-05-20)下的任务,**留给联调测试的缓冲缩到 5 天**(原本 12 天)
  - 下游"前后端联调"任务(2026-05-09 计划开始)依赖此完成 → 也得推 7 天?

我建议:
  [1] 仅改本任务,下游不动 → 联调可能压缩到 5 天,有风险
  [2] 改本任务 + 同步推下游"前后端联调" 7 天
  [3] 改本任务 + 整个 Beta 里程碑也延一周(从 2026-05-20 改到 2026-05-27)

你选哪个?(还是有别的方案)
```

### 类型 2 · 任务标阻塞

**触发场景**:用户说"X 卡住了"、"标阻塞"。

**skill 必做**(同 task-operations.md Op-M3 的引导式追问,这里强调下游分析):

5 件事一次问完:
1. **阻塞原因**:在等什么?等谁?
2. **预计解除时间**:1 天?3 天?不确定?
3. **是否影响下游**:有依赖此任务的其他任务吗?
4. **要不要 @ 帮手**:项目经理 / 关键人?
5. **是否需要平行替代方案**:能不能让其他人先做无依赖部分?

**对话样例**:
```
用户:推荐引擎接入卡住了

skill:已标阻塞 ✅。但需要补充几个信息让阻塞处理更高效:

1. 在等什么?(常见:等同事 / 等外部 / 等决策 / 等数据)
2. 预计什么时候能解除?(用于风险评估)
3. 这个任务卡住会影响下游哪些工作?
4. 要不要我现在 @ 项目经理或相关人?

你的回答:
```

(用户回答后,skill 把答案填到"风险与阻塞"字段 + 决定是否 @ 通知)

### 类型 3 · 改任务优先级到 P0

**触发场景**:用户说"X 改 P0"、"提到最高优先级"。

**skill 必做**:

1. **拉出当前所有 P0 任务**,看是否过载
2. **算这个新 P0 的"挤压"** — 同负责人是否同时多个 P0?同里程碑是否多 P0?
3. **追问**:把它提到 P0 是因为它真的紧急,还是别的 P0 优先级该降?

**对话样例**:
```
用户:把"测试集成"提到 P0

skill:可以,但提醒一下当前 P0 任务情况:

📊 现有 P0 任务({count} 个)
- 集成测试(测试组,5/21 开始)
- 灰度上线(测试组,6/06 开始)

⚠️ 加上 "测试集成"(测试组)→ **测试组手上 3 个 P0,且时间窗紧**

要怎么处理?
  [1] 确认提到 P0 — 测试组 3 个 P0 同时跑(可能加班)
  [2] 改提到 P1 — 给测试组喘息空间
  [3] 提到 P0 + 把"灰度上线"降 P1 — 因为灰度可以等 1 天

你选?
```

### 类型 4 · 加新任务插入到现有项目

**触发场景**:用户说"加一个 X 任务"。

**skill 必做**:

1. **问父任务归属**(避免任务漂浮无层级)
2. **看这个父任务的容量**(已有几个子任务?加一个会不会让它太宽?)
3. **看负责人** — 同时间该负责人手上几个任务?
4. **追问**:这个新任务是从哪挤的?有没有挤掉别的?

**对话样例**:
```
用户:加一个"标签性能压测"任务,负责人小张,5 月 25 日做

skill:确认下结构再加:

🔍 我推断它应该归到:
  - 里程碑:GA 发布(因为时间在 5/21-6/15 之间)
  - 工作包:测试与回归(测试相关)

❓ 问 4 件事:
  1. 父任务对吗?还是有别的归属?
  2. 小张当前手上有 4 个任务在 5/15-5/30 范围,加这个会不会过载?
  3. 这个任务挤掉了什么?(还是纯新增?)
  4. 计划完成日期 5 月 28 日(给 3 天合理吗,还是改成 5 月 30 日?)

回答完我再加。
```

## 不要做的事

- ❌ **不要无脑改字段** — 用户说"改 X" → 你直接改完汇报"已改"。这是被动工具行为
- ❌ **不要给一长串选项让用户晕** — 每次最多 3 个选项 + "或别的方案"兜底
- ❌ **不要分析过度** — 简单的小修改(改任务名、加一行最近更新)不需要连锁分析
- ❌ **不要假设用户知道下游影响** — 即使用户经验丰富,数据级别的分析仍有价值

## 跟其他文件的关系

| 文件 | 角色 |
|---|---|
| **本文件** | 给 skill 看的"分析方法论" — 怎么看连锁、怎么追问 |
| `task-operations.md` | 具体 Op 的执行步骤(改字段的命令) |
| `pm-onboarding.md` | 全景式 brief(本文件聚焦"单个变化的连锁") |

**关键区别**:
- task-operations = "**怎么改**"(命令、字段值、API 调用)
- task-cascade-effects = "**改之前看什么**"(分析、追问、决策)

两者**配合用**:用户说"改 X" → 先按 cascade-effects 分析 → 用户确认方案 → 按 task-operations 执行。
