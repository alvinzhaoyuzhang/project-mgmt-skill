# 日常任务操作 · 操作手册

> 这是 skill **每天**最常用的功能。当用户(成员或 PM)要让 skill **替他改/查任务**时读这个文件。
>
> 与 `pm-daily-guide.md` / `member-daily-guide.md` 的区别:
> - daily-guide = "我自己怎么用这个空间"(给思维方式引导)
> - **task-operations.md = "skill 帮我去飞书改任务/查任务"(skill 真动手)**

## 通用操作流程(skill 看这里)

每个任务操作走 4 步:

1. **识别上下文**
   - 用户身份:`lark-cli auth status` → 解析 JSON 里的 `userOpenId` 字段
   - 项目空间:看 `~/.pm-skill/registered_bases.json`,多空间时让用户指明
   - 涉及哪张任务表:Base 内可能多张,从用户语义判断

2. **找任务**
   - 用户用任务名说话(模糊),不会给 record_id
   - 用 `+record-search` 找候选:
     ```bash
     lark-cli base +record-search --base-token <base> --table-id <tid> \
       --json '{"keyword":"实现登录接口","search_fields":["任务名称"]}'
     ```
   - **拿 record_id**:从响应的 `data.record_id_list` 提取(注意:`record-search` / `record-list` 的 `data.data` 数组**不含 record_id**,必须看 `record_id_list`)
   - 多个匹配:列出来让用户选 [1] / [2] / [3]
   - 0 个匹配:告诉用户"没找到包含 X 的任务",问是不是写错或要不要搜其他词

3. **预览改动**
   - 列出**改什么字段**、**新值是什么**、**当前值是什么**
   - 等用户说"是 / 确认"才执行

4. **执行 + 回执**
   - 用 `+record-batch-update`:
     ```bash
     lark-cli base +record-batch-update --base-token <base> --table-id <tid> \
       --json '{"record_id_list":["<rid>"],"patch":{"进度":75,"最近更新":"..."}}'
     ```
   - 成功后告诉用户:"已更新 X 任务的 进度=75 / 最近更新=..."

## 操作清单 · 成员视角

### Op-M1 · 更新任务进度 + 最近更新

**用户输入示例**:
- "我的'实现登录接口'做到 80% 了,顺便填下最近更新"
- "把'写单测'进度改成 50,最近更新写'函数级测试完成,正在写集成测试'"

**字段 patch**:
```json
{
  "进度": 80,
  "最近更新": "2026-04-27 完成了 ... 下一步 ... 需要 ..."
}
```

⚠️ "最近更新"建议格式:**做完啥 / 下一步 / 需要谁**(三句话)。如果用户只给一句话,可以建议补充但不强求。

**进度字段是 number 类型**:范围 0-100,直接传整数。

### Op-M2 · 标记任务完成

**用户输入示例**:
- "我的'写部署文档'做完了,链接是 https://wiki.../doc/abc123"
- "X 任务完成,交付物是 PR https://github.com/.../pulls/456"

**字段 patch**:
```json
{
  "状态": "已完成",
  "实际完成日期": <今天 13 位毫秒时间戳>,
  "进度": 100,
  "交付物": "https://wiki.../doc/abc123"
}
```

⚠️ **必填"交付物"**。用户没给链接时,提示:"交付物字段建议填上(让 PM 验收时能直接点),你有链接吗?"

### Op-M3 · 标阻塞 + 填风险(引导式 5 问)

**用户输入示例**:
- "我的'部署测试环境'卡住了,等运维给环境"
- "X 阻塞了,后端没给我接口定义"

#### 步骤 1 · 标阻塞(立即执行,不等用户答完所有问题)

```json
{ "状态": "阻塞" }
```

#### 步骤 2 · 立即引导用户补 5 个信息(关键!这是引导式追问)

**严禁**只填用户给的零散信息就完事。skill 必须问全 5 个,然后把答案合并写入"风险与阻塞"字段:

```
✅ 已标阻塞。但**为了让阻塞处理更高效**,我需要你补 4 个信息:

1️⃣ 在等什么?(选一个最贴近的)
   [a] 等同事(具体谁?)
   [b] 等外部资源(数据 / API / 服务?)
   [c] 等决策(谁拍板?)
   [d] 等环境就绪(测试环境 / 部署 / 等)
   [e] 别的:你描述

2️⃣ 预计什么时候能解除?
   (1 天 / 3 天 / 1 周 / 不确定 — 不确定本身也是有用的信号)

3️⃣ 这个任务卡住会影响下游哪些工作?
   (例:"卡住后下个工作包没法开始" — 让 PM 知道连锁影响)

4️⃣ 要不要我现在 @ 项目经理或关键人通知一下?
   告诉我对方姓名,我帮你发飞书消息

回答这 4 个,我把它们整合成专业的"阻塞描述"写入字段,顺便决定要不要发通知。
```

#### 步骤 3 · 用户答完后,合成"风险与阻塞"字段值

```
[等同事-后端组长李四] 等待 /v2/users API 字段定义。
预计 2 天内解除(对方在赶 V1.5 发布,周三后能接此事)。
影响下游:本任务卡住会让"前端联调"也推迟 2 天。
已 @ 李四(2026-04-29 14:00 飞书消息),也已 cc PM 张三知情。
```

filed update:
```json
{
  "状态": "阻塞",
  "风险与阻塞": "<上面合成后的描述>",
  "最近更新": "<YYYY-MM-DD> 标阻塞 — 等待 ..."
}
```

#### 步骤 4 · 引导深一层(可选)

如果阻塞会让里程碑 / 项目延期,提示用户考虑改时间表:

```
ⓘ 这个阻塞预计 2 天解除,加上本任务剩余 3 天,实际完工 ≈ 5 天后(2026-05-04)。
   原计划 4-30 完成,现在可能延 4 天。
   要不要现在就改"计划完成日期"?(或等阻塞确实解除后再改也行)
```

(完整下游影响分析见 `references/task-cascade-effects.md`。)

### Op-M4 · 改截止日期 / 加协作人

**用户输入示例**:
- "X 任务的截止改成下周五"
- "把李四加进 X 任务的协作人"

**字段 patch**:
- 改日期:`{"计划完成日期": <13 位毫秒时间戳>}`
- 加协作人:**先 record-get 拿当前协作人列表 + 加新人 + 再 update**(协作人是 user 多选字段,update 是全量替换不是 append)

```json
{"协作人": [{"id":"ou_xxx_old"}, {"id":"ou_yyy_new"}]}
```

## 操作清单 · PM 视角

### Op-PM1 · 跨项目状态查询

**用户输入示例**:
- "我所有项目里有哪些 P0 任务卡住了?"
- "现在哪些任务延期超过 3 天?"

**实现**:
1. 从 `registered_bases.json` 拿用户的所有空间
2. 对每个空间,跑 record-search 或 record-list 加 filter:
   - "P0 卡住"= filter `优先级=P0-紧急 AND 状态=阻塞`
   - "延期 3 天+" = filter `状态=延期 AND 计划完成日期 < TODAY-3天`
3. 汇总成清单展示给用户

⚠️ 跨多个 Base 时**串行查询**,飞书 API 不允许并发。

### Op-PM2 · 重新分配任务

**用户输入示例**:
- "把张三的'前端组件'转给李四"
- "X 任务的负责人改成赵五"

**流程**:
1. 找张三的 open_id(用 `lark-cli contact user.search` 搜姓名)
2. 找李四的 open_id
3. 找张三名下的"前端组件"任务(用 record-search 关键词 + 自己加 filter `负责人=ou_zhangsan`)
4. 多个匹配让用户选哪个具体任务
5. patch:
   ```json
   {"负责人": [{"id":"ou_lisi"}]}
   ```
6. **告知用户**:任务改了负责人后,飞书会自动通知新负责人(让用户知道有这个副作用)

### Op-PM3 · 批量加任务(引导式 — 加之前先问连锁)

**用户输入示例**:
- "在 Q3 项目里加 3 个测试任务,负责人都是赵五,父任务是测试里程碑"
- "新增 X 任务"

#### 步骤 1 · 加之前必问 4 件事(引导式 — 防止任务漂浮)

skill 不能用户说"加 X" 就直接加,**必须先问连锁影响**(详见 `references/task-cascade-effects.md` 类型 4):

```
要加 {n} 个新任务,先确认 4 件事:

1️⃣ **父任务归属**:这些任务挂在哪个里程碑 / 工作包下?
   (我从描述推断是 [测试里程碑],对吗?)

2️⃣ **负责人容量**:负责人 [赵五] 当前手上有 {existing_count} 个任务,
   加这 {n} 个会不会让 ta 过载?(看时间窗 [start, end] 范围内任务量)

3️⃣ **挤压评估**:这些是纯新增,还是从其他工作里调整出来的?
   - 纯新增 → 看是否影响里程碑节奏
   - 调整 → 那原计划任务怎么办?

4️⃣ **里程碑容量**:目标父里程碑下已经有 {existing_subtask_count} 个任务,
   加这 {n} 个会不会让里程碑太宽?(原则:每层 3-7 个最合适)
```

#### 步骤 2 · 用户确认后批量加

```bash
lark-cli base +record-batch-create --base-token <base> --table-id <任务表> --json '{
  "fields": ["任务名称","任务级别","父任务","所属项目","负责人","状态","进度","计划完成日期","任务保密等级"],
  "rows": [
    ["X 任务", "📋 二级工作", [{"id":"<父 rid>"}], [{"id":"<项目 rid>"}], [{"id":"ou_zhao"}], "未开始", 0, <ms 时间戳>, "L2 常规"],
    ["Y 任务", "📋 二级工作", ...],
    ...
  ]
}'
```

3. 提示用户:WBS 编号要不要也帮你填?如果要,用父任务 WBS+顺序号

### Op-PM4 · 调整优先级(引导式 — 提到 P0 时必问"挤压")

**用户输入示例**:
- "把'修复登录 bug'优先级提到 P0"
- "X 任务优先级改 P1"

#### 步骤 1 · 改之前看一眼当前 P0 状态(只对"提到 P0"做,普通调整不需要)

如果用户提到 **P0**,skill 必拉所有当前 P0 任务给用户看,并算挤压风险(详见 `references/task-cascade-effects.md` 类型 3):

```
你要提"X"到 P0,先看下当前 P0 状态:

📊 当前 P0 任务({count} 个)
- 集成测试(测试组,5/21 开始)
- 灰度上线(测试组,6/06 开始)

⚠️ 提到 P0 后影响:
  - 测试组同时手 3 个 P0,且时间窗紧
  - "X" 计划完成日期是 5/15,在另两个 P0 之前

要怎么处理?
  [1] 确认提到 P0(测试组 3 个 P0 同时跑)
  [2] 改提 P1(给测试组喘息)
  [3] 提到 P0 + 把"灰度上线"降 P1(因为可以等 1-2 天)
  [4] 别的想法?

回我数字或方案。
```

#### 步骤 2 · 普通优先级调整(P1/P2/P3 之间)

直接改,不需要追问(影响小):

```json
{"优先级": "P1-高"}
```

⚠️ 选项值固定:P0-紧急 / P1-高 / P2-中 / P3-低,不要写"P0"或"紧急"。

### Op-PM5 · 查询某成员工作量

**用户输入示例**:
- "张三现在手上多少任务?"
- "看下李四这周到期的任务"

**实现**:
1. 找张三 open_id
2. record-search 加 filter `负责人 = ou_zhangsan AND 状态 != 已完成`
3. 展示数量 + Top 5 任务名

### Op-PM6 · 同步项目保密等级到所属任务

**何时触发**:用户在飞书 UI 或通过 skill 改了**项目主表**的「保密等级」字段,**该项目下所有任务的「任务保密等级」需要回写**。

**用户输入示例**:
- "我刚把 'V2.5 上线' 项目的保密等级从 L2 改成 L3,把任务也同步下"
- "项目主表 X 项目的保密等级改了,请 cascade 到任务"

**实现**:跑 cascade 脚本

```bash
python3 $SKILL_ROOT/scripts/_sync_secrecy.py \
  --base-token <base> \
  --project-name "<项目名>"
# 加 --dry-run 预览待改条目
```

脚本会:
1. 读项目主表当前「保密等级」
2. 找出该项目所有任务
3. 对「任务保密等级」字段不一致的批量更新

**为什么不用 lookup 自动同步?**:飞书的角色权限 filter **不支持 lookup 字段**,所以「任务保密等级」必须是 select 实体字段,由 skill 主动维护同步。详见 `references/feishu-permission-limits.md`。

### Op-PM7 · 项目状况全景询问(主动 brief)

**用户输入示例**(模糊全景式):
- "我刚回项目,看下状况"
- "X 项目什么情况"
- "给我个 overview"

**特殊处理**:**不要反问"看什么具体维度"**,而是直接跑 4 层 brief:

1. Layer 1 项目核心数字(完成率 / 里程碑达成 / 阻塞数 / 延期数 / P0 剩余)
2. Layer 2 最近 7 天关键事件(完成的里程碑 / 任务,新增的阻塞 / 延期)
3. Layer 3 3 件最该处理的事(从严重风险 + 临期 P0 + 待到期里程碑里挑 Top 3)
4. Layer 4 一句话引导:"你想从哪一项入手?"

完整流程 + 对话样例:**[references/pm-onboarding.md](pm-onboarding.md)**

### Op-PM8 · 项目延期重排(全局影响版)

**用户输入示例**:
- "X 项目要整体延一周"
- "Beta 发布推迟到 6 月"
- "我想把里程碑 1.2 的日期都往后推"

**这不是单任务延期(Op-M4),是项目级时间表重新规划。** skill 必做 4 步:

#### 步骤 1 · 拉出整个项目甘特

调 `feishu_bitable_app_table_record` action=list 任务表,filter 该项目,按 WBS 升序,展示:

```
项目"X"当前甘特(关键节点):
  1.1 需求冻结(已完成 4-15)
  1.2 Beta 发布(进行中,计划 5-20)
    1.2.1 后端开发(进行中,计划 5-15)
    1.2.2 前端开发(进行中,计划 5-18)
  1.3 GA 发布(未开始,计划 6-15)
  1.4 上线复盘(未开始,计划 6-30)
```

#### 步骤 2 · 问用户"延一周"是延谁?

```
你说的"延一周"是哪种?

  [1] 整个项目所有任务集体推 7 天(包括已完成的不动 + 未完成的全推)
  [2] 仅推未完成的里程碑 / 任务(已完成保留)
  [3] 仅推某个特定里程碑(那其他里程碑动不动?)
  [4] 别的方案?(详细描述)

(每种方案产生不同的批量更新,差别很大,不能凭"延一周"含糊处理。)
```

#### 步骤 3 · 用户确认方案后,展示**待改的具体清单**(必 preview)

```
将批量改 {n} 条记录的"计划完成日期":
  - 1.2 Beta 发布:5-20 → 5-27
  - 1.2.1.2 推荐引擎接入:5-15 → 5-22
  - ... ({n} 条)

确认?(回 "确认" 我就批量改)
```

#### 步骤 4 · 批量改 + 写"最近更新"留痕

每条任务的"最近更新"加一行:

```
2026-04-29 项目级延期 1 周 — 因 [原因](从用户输入提取)
```

## 公用操作

### 加新任务

```json
{
  "任务名称": "...",
  "任务级别": "📋 二级工作",
  "父任务": [{"record_id": "<父任务 rid>"}],
  "负责人": [{"id": "<ou>"}],
  "状态": "未开始",
  "进度": 0,
  "优先级": "P2-中",
  "计划完成日期": <时间戳>,
  "WBS编号": "1.2.3.4",
  "所属项目": [{"record_id": "<项目主表 rid>"}],
  "任务保密等级": "<从所属项目的'保密等级'同步>"
}
```

⚠️ **最少必填**:任务名称、任务级别、所属项目、负责人(如果是叶子任务)

⚠️ **任务保密等级必须同步**:本字段由 skill 写入时主动同步,不是 lookup。流程:
1. 创建前先读所属项目主表那行的「保密等级」字段
2. 把值原样塞进新任务的「任务保密等级」字段
3. 不要让用户手填,也不要留空
4. 同 Op-PM3 批量加任务时,所有 record 的「任务保密等级」都要带上

### 改任务名 / 任务描述

```json
{"任务名称": "新名字", "任务描述": "新描述"}
```

### 改父任务(调整层级)

```json
{"父任务": [{"record_id": "<新父任务 rid>"}], "WBS编号": "<新 WBS>"}
```

⚠️ 改父任务后**WBS 编号要手动改**(skill 不自动算,会和别的任务冲突)

### 改"所属项目"(任务转项目)

```json
{
  "所属项目": [{"record_id": "<新项目主表 rid>"}],
  "任务保密等级": "<新项目的保密等级,重新读一次>"
}
```

⚠️ 任务转项目时**两个字段必须一起改** — 因为「任务保密等级」是 skill 维护的 select 字段(见 `feishu-permission-limits.md`),所属项目变了它必须跟着变,否则该任务的可见性会错。

## 常见歧义处理

### 用户没指明哪个项目

```
当前账号下有 3 个项目空间:
  [1] 项目管理中心·产品研发
  [2] 项目管理中心·客户交付
  [3] 项目管理中心·行政流程

要在哪个空间操作?
```

### 用户没指明哪张任务表(B1 多任务表)

```
该空间内有 4 张任务表:
  [1] 任务表·Q3新版本上线
  [2] 任务表·客户A交付
  ...

X 任务在哪个里?
```

### 任务名匹配多个

```
找到 3 个匹配"实现接口"的任务:
  [1] T-0042 · WBS 1.2.3 · 实现登录接口 · 张三
  [2] T-0058 · WBS 1.3.1 · 实现支付接口 · 李四
  [3] T-0083 · WBS 1.4.2 · 实现导出接口 · 赵五

要操作哪个?
```

### 任务名匹配 0 个

```
没找到包含"X"的任务。可能:
  • 名字写错(常见:"接口" vs "API"、中英文不一致)
  • 该任务已被删除或归档

要不要换个关键词?或我列出你最近 7 天的任务?
```

## 不要做的事

1. **不要在用户没明确意图时主动改任务**
   - 错:用户说"我做完了 X" → skill 直接改成"已完成"
   - 对:用户说"我做完了 X" → skill 列出要改的字段(状态/实际完成日期/进度/交付物?),问"对吗?交付物链接给我"

2. **不要批量改不可逆字段**
   - 一次改 10+ 任务的"状态"或"负责人"前,让用户**逐条确认**或 dry-run 看一遍

3. **不要忽略权限**
   - 成员只能改自己负责的任务。如果用户(身份是成员)说"把张三的任务转给李四",拒绝并解释:"你只能改自己的任务,这个动作要让 PM 来做"

4. **不要走裸 API**
   - 永远用 `lark-cli base +xxx` 包装命令,不要 `lark-cli api /open-apis/bitable/v1/...`

## 跟 lark-base skill 的边界

如果用户的诉求**只是飞书 Base 的通用操作**(没有项目管理上下文),路由到 `lark-base` skill,不要在本 skill 内自己处理。

例:
- "帮我看下 X Base 有哪些字段" → lark-base skill
- "帮我在我的项目管理空间的 X 任务里改进度" → 本 skill(场景 C)
