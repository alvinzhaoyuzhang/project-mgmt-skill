## 阶段 A

**目标**:欢迎用户,自动检查环境,确认账号。

### 屏 1 · 欢迎

```
✨  项目管理中心 · 安装向导
─────────────────────────────────────────────

这个 skill 会在你的飞书里创建一套项目管理模板,
内容包括:
   • 项目主表与任务表
   • 综合看板(自动生成进度图表)
   • 3 级权限角色(超管 / 项目经理 / 成员)

⏱  约 5 分钟(网络慢可能 10 分钟)
🛡  每步都会先 preview 再让你确认
↩  随时按 Ctrl+C 暂停,下次跑会继续

按 Enter 开始,或输入 q 退出 ▸
```

**用户输入**:
| 输入 | 行为 |
|---|---|
| Enter / 空 / `y` / `是` | → 屏 2 |
| `q` / `退出` | 优雅退出,无副作用 |
| 其他 | "未识别,按 Enter 开始或 q 退出" 重新等待 |

### 屏 2 · 自动检查 + 账号确认

skill 调用 Bash 跑检查,把结果一次显示:

```bash
# 内部检查
lark-cli --version  # 期望 ≥ 1.0.18
lark-cli auth whoami  # 期望返回登录态
```

**正常情况(99% 走这路径)**:

```
🔍  正在检查环境...

   ✅  网络可达 feishu.cn
   ✅  OpenClaw 飞书插件 已安装(v1.0.18)
   ✅  飞书已登录:小王  /  你的租户名

⚠️  接下来会用此账号在飞书内创建表格,
   你将默认成为新空间的"超级管理员"。

继续? [Enter 继续 / 切换账号 / q 退出] ▸
```

**异常情况**:

| 检查项失败 | 显示文案 | 用户选项 |
|---|---|---|
| 飞书插件未装 | `❌ 未安装 OpenClaw 的飞书插件` + 安装命令 `npx -y @larksuite/openclaw-lark install` | [r 重试] [q 退出] |
| 插件版本过低 | `⚠️ 版本 < 1.0.18,请运行 npx -y @larksuite/openclaw-lark install --force` | [r 重试] [q 退出] |
| 飞书未登录 | `❌ 飞书未登录` + 提示去 OpenClaw 重新连接 | [r 重试] [q 退出] |
| 用户输 "切换账号" | "请回到 OpenClaw 重新绑定飞书账号,完成后输 r 重试" | [r] [q] |

### 阶段 A 完成条件

- 飞书已登录 + 插件就位 + 用户确认账号
- 写入状态文件 `~/.pm-skill/wizard.state.json`,标记 stage=A 完成

## 阶段 C

**目标**:判断用户是首装还是已有空间,路由到正确路径。

### 检测逻辑

```python
# 伪代码
local_bases = read_json("~/.pm-skill/registered_bases.json")
online_bases = lark-cli base +base-list, filter name LIKE "项目管理中心·*"

if not local_bases and not online_bases:
    → 屏 1(第一次)
elif local_bases and len(local_bases) == 1:
    → 屏 2(已有 1 个空间菜单)
elif local_bases and len(local_bases) > 1:

## 阶段 D

**目标**:问 2 个问题,推断架构方案,告诉用户结果。

### 屏 1 · 团队规模(Q4)

```
📊  规模规划 (1/2)

你团队大约多少人?

   [1] < 5 人(小团队 / 一个人玩)
   [2] 5-15 人(中型团队)
   [3] 15+ 人(大团队 / 多团队联合)

ⓘ 人数指你计划用这个空间的人,不一定是公司总人数

▸ 选 1-3:
```

### 屏 2 · 项目并发(Q5)

```
📊  规模规划 (2/2)

你打算同时跑几个项目?

   [1] 1-3 个(单线作战 / 一个产品)
   [2] 4-10 个(多线并进)
   [3] 10+ 个(项目密集)

ⓘ "项目"指独立的工作流(一个产品、一个客户、一次交付),
   不是任务

▸ 选 1-3:
```

### 屏 3 · 推荐方案

skill 内部根据 Q4 × Q5 推断架构(详见 [architecture-decision.md](architecture-decision.md))。

**例 · 5-15 人 + 4-10 项目** → 推荐"多任务表":

```
✨  根据你的回答,推荐方案:

   团队规模:5-15 人
   项目并发:4-10 个

   推荐:多项目独立任务表
   ✓  在 1 个空间内管所有项目
   ✓  每个项目独立任务表 + 独立看板
   ✓  PM 一个空间内切换,不切来切去
   ✓  成员只看到自己项目的任务

是否按此方案继续?
   [是] 继续
   [看其他方案] 显示 3 选 1
   [退出]
▸ 
```

```

  [1/9] ✅  创建项目空间                    (3s)
  [2/9] ✅  建项目主表(11 字段)            (12s)
  [3/9] ✅  建任务表模板(30 字段)          (35s)
  [4/9] ✅  建速查卡 + 填 34 条规范        (8s)
  [5/9] ⏳  正在启用高级权限...
  [6/9] ⏸  待:创建 3 角色
  [7/9] ⏸  待:建项目主表 4 视图
  [8/9] ⏸  待:建任务表 9 视图
  [9/9] ⏸  待:建综合看板模板

### 失败处理

| 场景 | 自动处理 | 用户选项 |
|---|---|---|
| 网络抖动超时 | 自动重试 1 次,失败再询问 | [r 重试] [s 跳过] [a 中止] [u 撤销] |
| 字段创建冲突 | 询问 | 同上 |
| 权限不足 | 暂停,提示用户找 IT | [a] [u] |
| 用户 Ctrl+C | 保存状态,优雅退出 | 下次续上 |

`[u] 撤销` = `lark-cli base +base-delete --base-token <new_base> --yes`,完全清理。

## 中断保护

### 状态文件位置

`~/.pm-skill/wizard.state.json`

### 结构

```json
{
  "version": "1.0",
  "started_at": "2026-04-25T18:30:00+08:00",
  "current_stage": "D",
  "completed_stages": ["A", "C"],
  "answers": {
    "Q4_team_size": "5-15",
    "Q5_project_count": "4-10",
    "_inferred_arch": "B1",
    "Q6_domain": "产品研发",
    "space_name": "项目管理中心·产品研发"
  }
}
```

### 写入时机

每完成一个 Q 就 atomic write(临时文件 + rename 防止部分写入)。

### 续跑

下次跑 `pm init`,先检测状态文件:

```
⚠️  发现上次未完成的安装(中断在 阶段 D)
   你之前回答到:团队规模 = 5-15 人

→ [继续上次] [重新开始] [取消]
```

### 状态文件损坏

```
⚠️  上次状态文件损坏,无法续跑。
   是否清除并重新开始? [是 / 否]
```

清空命令:`rm ~/.pm-skill/wizard.state.json`(用户主动 `pm init --reset`)。
