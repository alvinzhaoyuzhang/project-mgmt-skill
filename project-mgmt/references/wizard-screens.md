## 对话节奏 · 批量提问 vs 一问一答

**这一节决定 skill 怎么跟用户对话,LLM 必须严格遵守,不能随机切换。**

### 默认模式:**可批量** + **关键决策点必须独立**

| 风险级 | 问题清单 | 模式 | 说明 |
|---|---|---|---|
| 🟢 **低风险** | 阶段 D Q4 团队规模、阶段 E Q6 业务方向、阶段 E Q7 空间名、阶段 F Q8 任务层级 | **可批量** | 答错影响小,改起来便宜。可以一次性列出来让用户在一个 message 里答完 |
| 🟡 **中风险** | 阶段 D Q5 项目并发(普通值 1-3 / 4-10)、阶段 F Q9 保密分级 | **可批量,但必须用最新文案展示** | 答案有歧义可能,**必须把 Q5 / Q9 文案完整带着澄清说明展示给用户**(不要自己压缩),让用户看到 "这只是这一个空间的项目数" 这种消歧义提示 |
| 🔴 **必须独立** | 阶段 D · Q5=10+ 安全网追问、阶段 G · Preview、阶段 J · 加项目时的 G+ 4 题、阶段 I · 验证后回执 | **不可批量,必须一屏一问** | 涉及条件分支 / 写操作前确认 / 真实业务输入 — 漏问任何一个安全网都会引入 silent bug |

### 具体规则

1. **进入 wizard 时先问一句节奏偏好**(可选,默认走批量):

   ```
   👋 我有 6 个简短问题,然后帮你建空间。
   你想:
     [1] 一次性列出来,你一段话答完(默认,更快)
     [2] 一个一个问(更稳)

   ▸ 选 1-2 或直接答(默认 [1]):
   ```

2. **批量模式下,必须把"关键消歧义文案"原样带上**

   错误示范(LLM 自己压缩):
   > "请告诉我:1) 团队人数 2) 项目并发数 3) 业务方向 4) 复杂度"
   >
   > → 用户答"管 20 多个项目" → LLM 直接归为 [3] 10+ → **绕过我们的 Q5 文案澄清,直奔 B2 多空间架构**

   正确示范(LLM 完整展示带澄清的文案):
   > "请按以下顺序答 4 题:
   >
   > **Q4 团队规模**: ...
   >
   > **Q5 项目并发** — 注意:**这一个空间**里大概要装多少个项目?(其他业务方向可另建空间)
   >    [1] 1-3 / [2] 4-10 / [3] 10+
   >    ⓘ 不是问'你团队总共管多少',只算这一个空间...
   >
   > **Q6 业务方向**: ...
   >
   > **Q8 任务层级**: ..."

3. **触发 🔴 红线情况时,LLM 立即切回独立提问**

   - **Q5 答了 10+** → **必须**再独立问安全网("是不是同方向?"),哪怕用户已经在批量答里给了所有问题
   - **Q9 选了分级** → 在 Preview 之前必须独立提示"加项目时还会问每个项目的等级"
   - **进入阶段 G Preview** → 永远独立屏(不能跟其他东西批量合并)
   - **加项目 G+ 4 题** → 永远独立(项目名 / PM / 成员 / 保密等级 各自一屏)

4. **如果用户在批量答中漏答某题**

   - LLM 不要假设默认值,**必须独立追问那一题**
   - 不要直接进入下一阶段

### 为什么这么设计

我们的 wizard 文案里加了多处"消歧义"和"安全网"(Q5 项目并发、Q6 业务方向、Q5=10+ 反问、Q9 保密分级、G+4 加项目时的保密等级)。这些**只在用户看到完整文案 + skill 触发条件分支时**才生效。

如果 LLM 自己批量压缩成"请答 6 题",这些防护就全失效了 — 用户经常会因此被推到错误的架构(如把 5 个跨方向项目误推 B2),或在加项目时漏问保密等级。

**节奏这事儿不是"风格选择",是"流程契约"**。LLM 必须按这个章节执行。


### 屏 2 · 自动检查 + 账号确认

skill 调用 Bash 跑检查,把结果一次显示:

```bash

```

**正常情况(99% 走这路径)**:

```

   ✅  网络可达 feishu.cn
   ✅  飞书 CLI(lark-cli) 已安装(v1.0.9)
   ✅  飞书已登录:小王  /  你的租户名

   你将默认成为新空间的"超级管理员"。

继续? [Enter 继续 / 切换账号 / q 退出] ▸
```

**异常情况**:

| 检查项失败 | 显示文案 | 用户选项 |
|---|---|---|
| 飞书 CLI 未装 | `❌ 未安装 飞书 CLI`<br>**主动建议用户**:"我可以帮你装,你只需复制这一句给我:`帮我安装飞书 CLI:https://open.feishu.cn/document/no_class/mcp-archive/feishu-cli-installation-guide.md`"<br>用户同意后,skill 自己 Bash 跑 `npx @larksuite/cli@latest install` → 提示用户**重启 AI 工具** → 重启后跑 `lark-cli auth login`。详见 troubleshooting.md §1.1 | [让 AI 帮装] [手动装] [q 退出] |
| CLI 版本过低 | `⚠️ 版本 < 1.0.9,请重新跑 npx @larksuite/cli@latest install(@latest 会自动取最新版),装完后重启 AI 工具` | [r 重试] [q 退出] |

| 用户输 "切换账号" | "跑 `lark-cli auth logout` 后再 `lark-cli auth login` 用其他账号,完成后输 r 重试" | [r] [q] |

### 阶段 A 完成条件

- 飞书已登录 + lark-cli 就位 + 用户确认账号
- 写入状态文件 `~/.pm-skill/wizard.state.json`,标记 stage=A 完成

## 阶段 D

**目标**:问 2 个问题,推断架构方案,告诉用户结果。

### 屏 1 · 团队规模(Q4)

```
📊  规模规划 (1/2)

你团队大约多少人?

   [1] < 5 人(小团队 / 一个人玩)
   [2] 5-15 人(中型团队)
   [3] 15+ 人(大团队 / 多团队联合)

ⓘ 人数指你计划用这个空间的人,不一定是公司总人数

▸ 选 1-3:
```

### 屏 2 · 项目并发(Q5)

```
📊  规模规划 (2/2)

**这一个空间**里大概要装多少个项目?

   [1] 1-3 个 — 单条线 / 一个产品 / 一组紧密相关的事
   [2] 4-10 个(主流) — 多条线并行,同一业务方向
   [3] 10+ 个 — 项目密集(通常说明该按业务方向拆多个空间)

ⓘ 关键:这里只算**这一个空间**会装的项目数,**不是你团队总共管多少**。
   - 你可能管 20 个项目,但分散在产品研发 / 客户交付 / 行政事务等不同方向
   - 建议**每个业务方向单独建一个空间**,回主菜单可继续建第二、三个空间
   - 例:产品研发 8 个项目 → 一个空间;客户交付 5 个 → 另一个空间;其余分别建

ⓘ "项目" = 一条独立工作流(一个产品 / 一个客户 / 一次交付),不是单个任务

▸ 选 1-3:
```

**Q5 = [3] 10+ 时的安全网**:

skill 必须**追问一次**(除非用户已明示),避免误推 B2:

```
你选了「10+ 个项目」 — 我想再确认下:

  这 10+ 个项目是不是都属于**同一个业务方向**?
  (比如都是产品研发,或都是客户交付)

  [1] 是 — 都是同方向,需要在一个空间里统一管
       └ skill 推荐 B2 多空间架构(每项目独立 Base)
  [2] 否 — 跨多个方向,可以拆
       └ skill 建议你**先建当前最主要方向的空间**,
          这次只填那个方向的项目数(可能就 4-10 而已),
          其他方向回主菜单单独建

▸ 选 1-2:
```

选 [2] 后 skill 跳回 Q5,让用户重新答**只针对主方向**的项目数。

```

是否按此方案继续?
   [是] 继续
   [看其他方案] 显示 3 选 1
   [退出]
▸ 
```

### 屏 2 · 保密分级(Q9)

```
🔒  保密分级 (2/2)

你的团队需要按项目设置不同的保密等级吗?(空间级开关)

  [1] 不需要 — 所有项目统一可见,简化权限
       └ 项目主表不会建"保密等级"字段
       └ 所有成员都能看所有项目

  [2] 需要 — 启用 L1-L4 四档分级
       └ 项目主表加"保密等级"字段(L1 公开 / L2 常规 / L3 敏感 / L4 机密)
       └ 加项目时再单独问"这个项目分到哪一级"
       └ 不同等级的项目对成员的可见性不同

ⓘ 这只是**空间级开关** — 决定整套分级机制是否启用。
   具体每个项目分到哪一级,后面 G+4 加项目时才问。
ⓘ 选 [1] 后续不能直接升 [2](等于重置权限)

▸ 选 1-2 (默认 [1]):
```

### 状态文件位置

`~/.pm-skill/wizard.state.json`

### 结构

```json
{
  "version": "1.0",
  "started_at": "2026-04-25T18:30:00+08:00",
  "current_stage": "D",
  "completed_stages": ["A", "C"],
  "answers": {
    "Q4_team_size": "5-15",
    "Q5_project_count": "4-10",
    "_inferred_arch": "B1",
    "Q6_domain": "产品研发",
    "space_name": "项目管理中心·产品研发"
  }
}
```

### 写入时机

每完成一个 Q 就 atomic write(临时文件 + rename 防止部分写入)。

### 续跑

下次跑 `pm init`,先检测状态文件:

```
⚠️  发现上次未完成的安装(中断在 阶段 D)
   你之前回答到:团队规模 = 5-15 人

→ [继续上次] [重新开始] [取消]
```

### 状态文件损坏

```
⚠️  上次状态文件损坏,无法续跑。
   是否清除并重新开始? [是 / 否]
```

清空命令:`rm ~/.pm-skill/wizard.state.json`(用户主动 `pm init --reset`)。
